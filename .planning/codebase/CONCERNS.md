# Concerns

## Product-Implementation Mismatch

- The repository docs describe a LangGraph-driven, durable, Postgres-backed workflow, but `app/graph/workflow.py` is a custom in-memory Python class.
- The live request path does not use SQLAlchemy models or sessions even though persistence models and migrations exist in `app/db/`.
- The current implementation is closer to a vertical prototype than the finished product described in `docs/specs/2026-03-30-investor-design.md`.

## Workflow Durability

- `app/main.py` initializes `app.state.workflow_store = {}`.
- `app/api/routes.py` stores paused workflow state only in that in-memory dict.
- Any process restart loses approval state and makes existing approval links unusable.
- This is the largest runtime reliability gap in the app.

## Approval Flow Integrity

- The workflow composes approval URLs as `http://localhost:8000/approval/{run_id}:approve` and `...reject` in `app/graph/workflow.py`.
- The approval route expects a signed token and calls `verify_approval_token`, so the URLs generated by the workflow do not match the verifier contract.
- Tests pass because route tests create signed tokens directly instead of exercising the links generated by the workflow.

## Persistence Drift

- `app/db/models.py` defines richer runtime records, but there is no code persisting runs, recommendations, approval events, or state transitions during normal API use.
- `app/db/session.py` is not imported by the API or workflow code paths.
- The app therefore lacks auditability despite having audit-oriented models.

## Configuration Drift

- `.env.example` points to Postgres, but `app/config.py` defaults to SQLite.
- Tests force SQLite memory mode in `tests/conftest.py`.
- This split is manageable for local dev, but it increases the chance that Postgres-specific issues stay hidden.

## Error Handling

- `GET /approval/{token}` assumes both token validity and run presence.
- Invalid tokens and unknown `run_id` values will currently surface as unhandled exceptions or framework-default 500 responses.
- The API lacks user-facing error semantics for expired, invalid, or stale approvals.

## Integration Thinness

- `send_console_email` does not deliver anything.
- `ResearchNode` accepts JSON from an injected LLM but does not integrate with Quiver or Alpaca clients.
- The “agent” and “workflow” naming suggest deeper capability than what is currently implemented.

## Operational Gaps

- No CI pipeline is present.
- No linting or static analysis is configured.
- No deployment or scheduling path exists for the “daily run” behavior in the product spec.
- No secrets-management or environment validation beyond Pydantic defaults is present.
